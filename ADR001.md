# ADR-001: Repository Strategy for Multi-Tenant AEMaaCS — Multi-Repo + Aggregator (Program-Wide)

## Context

The program is migrating multiple AEM tenant applications from Adobe Managed Services (AMS) to **AEM as a Cloud Service (AEMaaCS)**.

Key program constraints and operating realities:

1. **Single Cloud Manager Program / shared environments (Dev / Stage / Prod)**  
   * The deployed runtime in a given environment represents a **single customer codebase**.

2. **Cloud Manager repository constraint**  
   * Cloud Manager pipelines attach to a single Git repository per pipeline.  
   * Pipeline executions build and deploy artifacts from that repository into the targeted environment.

3. **Many tenants at program scale**  
   * The number of tenant applications is too large to operate as separate Cloud Manager Programs or environment sets per tenant.

4. **Tenant interdependencies**  
   * Tenants share platform components and may have cross-tenant runtime or API dependencies.

5. **AEMaaCS pipeline-first / immutable runtime model**  
   * Deployments provision fresh runtime containers.  
   * Runtime changes must occur through pipeline deployments rather than manual server-side changes.

Given these constraints, the program requires a repository strategy that:

* Scales to many tenants with clear ownership boundaries
* Supports composition into a single deployable codebase for Cloud Manager
* Reduces integration drift and “last-minute assembly surprises”
* Enables deterministic builds, traceability, and auditability
* Supports disciplined release and hotfix operations

---

## Problem Statement

We need a program-wide repository topology that balances:

* Tenant team autonomy and ownership clarity
* Controlled integration of shared platform modules
* Predictable assembly into a deployable artifact appropriate for Cloud Manager
* Operational stability in a shared AEMaaCS runtime

---

## Decision

Adopt a **Multi-Repo + Aggregator** repository strategy program-wide.

* Each tenant maintains its **own repository** containing tenant-owned code, configuration, and tests.
* A separate **Aggregator repository** acts as the deployment composition point.

The Aggregator repository serves two primary roles:

1. **Deployment Assembly**  
   * The Aggregator composes the full deployable codebase that Cloud Manager builds and deploys.

2. **Shared Platform Modules**  
   * Shared platform modules used by multiple tenants are maintained within the Aggregator repository.

This decision applies program-wide to all tenants participating in the AEMaaCS platform.

---

## Rationale

### 1. Scales better than a monorepo for many tenants

With many tenant teams, a monorepo increases:

* global coordination overhead
* review and permissions complexity
* CI contention and longer feedback cycles
* “noise” from unrelated changes

A multi-repo model preserves team autonomy and reduces unnecessary coupling.

---

### 2. Aligns with Cloud Manager constraints

Because:

* Cloud Manager pipelines operate against a single repository, and
* environments deploy a single composite customer codebase,

the system requires a clear composition point for the deployable artifact.

The Aggregator repository provides that controlled composition point.

---

### 3. Makes cross-tenant dependencies explicit

Shared platform modules represent the most common source of cross-tenant coupling.

Centralizing shared modules in the Aggregator repository:

* enables governance of shared component evolution
* reduces uncontrolled dependency drift
* provides a consistent location for compatibility validation

---

### 4. Supports deterministic builds and traceability

In a multi-repo model, deterministic composition is critical.

The Aggregator model enables deterministic builds by:

* selecting specific tenant revisions for each release
* producing a single deployable artifact for Cloud Manager
* enabling traceability between tenant commits, aggregator composition, and deployed releases

Release composition and runtime identity mechanisms are defined in **ADR-003**.

---

## Alternatives Considered

### Alternative 1: Monorepo (all tenants in a single repository)

**Pros**

* Atomic cross-tenant changes
* Simplified release identification (single commit SHA)
* No assembly step

**Cons**

* Governance and permissions complexity increases significantly
* CI contention grows with repository size
* Reduced tenant autonomy
* Higher coordination overhead

**Decision**

Rejected due to operational friction and scaling risks for many independent tenant teams.

---

### Alternative 2: Multi-repo without an Aggregator

**Pros**

* Maximum tenant autonomy

**Cons**

* No authoritative composition point
* Higher risk of integration drift
* Difficult to assemble a deployable codebase for Cloud Manager
* Reduced traceability and auditability

**Decision**

Rejected because it undermines release determinism and operational stability.

---

## Consequences

### Positive

* Clear tenant ownership boundaries
* Scales effectively across many teams
* Shared platform modules governed centrally
* Explicit and controlled deployment composition
* Supports deterministic release practices when combined with manifest discipline

### Negative

* Assembly risk exists without strict version pinning
* Aggregator repository becomes critical infrastructure
* Requires disciplined governance and automation to maintain build determinism

---

## Required Guardrails

To prevent the Aggregator from becoming an integration bottleneck, the program must enforce:

* **No floating dependency versions**
* **Pinned composition for any deployable build**
* **Release bill of materials (BOM) per production deployment**
* **Ownership controls for shared platform modules**
* **Aggregator-level integration testing**

Detailed build identity, versioning, and traceability mechanisms are defined in **ADR-003**.

---

## Decision

Use a **Multi-Repo + Aggregator** repository strategy program-wide, with shared platform modules living in the Aggregator repository and deterministic deployment composition enforced through controlled assembly practices.
