# ADR-002: Release Train Model for Multi-Tenant AEMaaCS (Aggregator-Based)

## Status
Proposed (ready for review)

## Date
2026-03-02

## Related Decisions
* ADR-001: Repository Strategy for Multi-Tenant AEMaaCS (Multi-Repo + Aggregator)
* ADR-003: Build, Versioning, and Runtime Identity Strategy for Multi-Tenant AEMaaCS

---

## Context

This ADR assumes the repository topology defined in ADR-001:

* Multi-repo tenant model
* Aggregator repository as the deployable composition point
* Shared Cloud Manager Program
* Shared environments: **Dev → SIT → Stage → Prod**
* A single composite deployable artifact per environment

Given this structure, we must define a release operating model that:

* Preserves **Stage as a credible production proxy**
* Minimizes integration drift across tenants
* Enables deterministic rollback
* Provides a safe but fast hotfix path
* Scales across many tenants with interdependencies
* Operates within Cloud Manager pipeline constraints

Ad-hoc release patterns from AMS are incompatible with this operating model.

---

## Problem Statement

In a shared-environment, composite-artifact model:

* Independent tenant releases can create cross-tenant integration instability.
* Stage loses credibility if artifact integrity is not preserved.
* Rollback must operate at the composite artifact level.
* Hotfixes must be fast without bypassing pipeline discipline.

We require a structured release mechanism that balances stability, speed, and auditability.

---

## Decision

Adopt a **time-boxed Release Train model** anchored in the Aggregator repository.

The Release Train will:

* Use an **Aggregator Release Branch** as the train system-of-record.
* Use a **Pinned Release Manifest** to lock the tenant revisions included in the train.
* Produce a **single composite artifact** per train.
* Promote that artifact through **Dev → SIT → Stage → Prod**.
* Provide a defined **Hotfix Lane** branching from the last production tag.

The manifest structure and runtime identity mechanisms are defined in **ADR-003**.

---

## Decision Points

### DP1: Release Operating Model

**Options**

* Ad-hoc / “release when ready”
* Continuous deployment per tenant
* Time-boxed Release Train

**Decision**

Time-boxed Release Train.

**Rationale**

* Reduces integration drift.
* Creates predictable stabilization windows.
* Preserves environmental integrity.
* Aligns with the composite artifact deployment model.

---

### DP2: Release Anchor

**Options**

* Integration branch across multiple repositories
* Aggregator release branch with pinned manifest

**Decision**

Aggregator release branch with pinned manifest.

**Rationale**

* Establishes a single source of truth for release composition.
* Enables deterministic rebuilds.
* Allows rollback by redeploying a prior tagged release.

---

### DP3: Manifest Strategy

**Options**

* Floating tenant versions
* Pinned manifest per train

**Decision**

Pinned manifest per train.

**Controls**

* No floating or “latest” versions.
* Manifest changes require pull request review.
* Each train produces a **release bill of materials (BOM)**.

Detailed manifest structure and tenant identity mechanisms are defined in **ADR-003**.

---

### DP4: Promotion Model

**Options**

* Rebuild artifact independently in each environment
* Build once and promote the same artifact across environments

**Decision**

Build once and promote.

**Promotion Path**

Dev → SIT → Stage → Prod

**Rationale**

* Preserves environment parity.
* Eliminates build drift between environments.
* Ensures Stage validation reflects the production artifact.

---

### DP5: Hotfix Strategy

**Options**

* Emergency deployments that bypass the train
* Controlled hotfix lane producing a composite artifact

**Decision**

Controlled hotfix lane.

**Hotfix Mechanics**

* Branch from the last production tag in the Aggregator repository.
* Apply minimal changes required to address the incident.
* Build a new composite artifact.
* Promote through **Dev → SIT → Stage → Prod** using an expedited validation path.
* Rollback remains the redeployment of the prior production tag.

---

## Release Train Operating Model

### Cadence (Example)

* 2-week cycle (tunable)
* Train phases:

1. Development window
2. Release cutoff
3. Stabilization and integration testing
4. Stage validation
5. Production release
6. Post-release monitoring and retrospective

---

### Inclusion Model

* Tenant changes are included only when their revision is pinned into the release manifest.
* If a tenant misses the cutoff, its change rolls to the next train.

---

### Governance Roles

Minimum governance roles include:

* **Release Manager**
* **Platform / DevOps Owner**
* **Tenant Technical Leads**
* **Program Manager**

---

### Release Readiness Criteria

A release candidate must satisfy:

* Composite artifact passes Cloud Manager validation gates
* Integration and smoke tests pass
* Production rollback target confirmed
* Release manifest and change log published

---

## Risks and Mitigations

### Risk: Aggregator integration instability

Mitigation:

* Continuous integration testing on the Aggregator mainline
* Early integration testing before release cutoff

---

### Risk: Rollback affects all tenants

Reality:

Rollback occurs at the composite artifact level.

Mitigation:

* Small, frequent trains
* Feature toggles where appropriate
* Strong Stage validation

---

### Risk: Pipeline congestion

Mitigation:

* Published release calendar
* Controlled promotion windows
* Limiting non-train deployments

---

## Consequences

### Positive

* Predictable release cadence
* Deterministic rollback
* Stage remains a credible production proxy
* Clear audit trail through release artifacts and manifests
* Reduced operational instability

---

### Negative

* Requires disciplined release cutoffs
* Tenant changes may occasionally defer to the next train
* Hotfix deployments still affect the composite artifact

---

## Decision

Adopt a **Release Train model** anchored in the Aggregator Release Branch with a pinned manifest, producing a single composite artifact promoted through **Dev → SIT → Stage → Prod**, with a controlled hotfix lane.
