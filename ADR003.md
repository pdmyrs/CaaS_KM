# ADR-003: Multi-Tenant AEMaaCS Build, Versioning, and Runtime Identity Strategy (Shell Aggregator)

## Status
Proposed (ready for review)

## Date
2026-02-02

## Decision Owner

## Context

TRP is migrating multiple AEM applications from Adobe Managed Services (AMS) to AEM as a Cloud Service (AEMaaCS).

In AMS, TRP operates a multi-tenant delivery model where each tenant team owns an independent Git repository and can deploy packages independently using Package Manager and TRP tooling (e.g., Raven). This enables rapid iteration and rollback patterns (e.g., uninstalling packages).

TRP tenants have runtime dependencies on other tenants. Example dependency chain:

* Enterprise Site bundle `enterprise-aem-core` imports Java packages from Compose bundle `trp-compose-core` and ECL bundle `trp-ecl-core`
* IID Core bundle `iid-aem-core` imports packages from Compose
* FAI bundle `fai-site.core` depends on IID Core, creating transitive dependencies across tenants

Compose (Beacon design system) changes frequently and, in AMS, can be redeployed multiple times per day to Stage while consumer teams validate and defects are discovered/resolved over multi-day cycles.

In AEMaaCS, TRP must adapt to Cloud Manager constraints, including:

* Immutable runtime deployments (no incremental runtime install/uninstall strategy)
* Pipeline validation gates
* Restrictions on repository-to-pipeline attachment
* Stage/Prod auto-versioning behavior that rewrites module versions
* **(Confirmed) Stage/Prod version rewriting also rewrites OSGi bundle versions**, distorting normal OSGi version signaling

TRP also has a security constraint:

* TRP Artifactory cannot be exposed to the public internet, preventing Cloud Manager from fetching private binary dependencies from TRP infrastructure.

TRP requires a long-term operating model that:

* Preserves multi-team independence at source control (per ADR-001)
* Enables reliable promotion across environments (Dev/SIT/Stage/Prod)
* Meets enterprise governance (traceability, approvals, rollback, hotfix)
* Explicitly addresses cross-tenant dependency risks
* Is defensible to architecture, risk, and security stakeholders
* Provides accurate runtime visibility into “what tenant version is running” despite Cloud Manager rewriting behavior

---

## Problem Statement

How can TRP support a multi-tenant repository model with runtime dependency relationships while deploying to AEMaaCS, given that Cloud Manager enforces single-repo pipeline inputs, immutable deployments, mandatory validations, security restrictions on dependency resolution, and Stage/Prod auto-versioning behavior that **rewrites Maven module versions and OSGi bundle versions**, invalidating Maven/OSGi version fields as stable per-tenant identifiers?

---

## Drivers and Constraints

### 1) Cloud Manager Single Pipeline Input Model
Cloud Manager pipelines attach to a single repository for a given pipeline. A pipeline execution builds and deploys from that repository. AEMaaCS does not support deploying arbitrary subsets of code from separate repositories into a shared runtime the way AMS Package Manager does.

### 2) AEMaaCS Ephemeral and Immutable Runtime
Deployments provision fresh runtime containers. Deployment must include the full required set of applications and configurations. Incremental package installs and runtime rollback via uninstall are not available as an operational strategy.

### 3) Cloud Manager Validation Gates
Cloud Manager enforces automated validation checks that can block deployments (code quality, security scanning, unit tests, runtime validations). Any failure blocks promotion and can therefore impact release schedules across all tenants included in the deployment unit.

### 4) TRP Artifactory Access Restriction
TRP Artifactory is not publicly reachable; Cloud Manager cannot fetch private Maven artifacts from TRP Artifactory. Any solution that relies on Cloud Manager resolving cross-tenant dependencies from TRP Artifactory is not feasible.

### 5) Cloud Manager Stage/Prod Auto-Versioning (Maven + OSGi)
For Stage and Prod full-stack pipelines, Cloud Manager runs Maven version rewrite operations across the reactor (e.g., via the Maven Versions Plugin with broad selectors). There is no supported mechanism to opt out modules from this rewrite.

**Confirmed consequence:** Stage/Prod rewriting also distorts normal OSGi version signaling (bundle versions), which means:

* Per-tenant Maven artifact versions do not remain stable in Stage and Prod builds
* OSGi `Bundle-Version` does not remain a reliable per-tenant semantic identifier in Stage/Prod
* Any design that assumes stable per-module Maven or OSGi versions at deploy time is invalid

### 6) Environment Model Requirement
TRP maintains Dev, SIT, Stage, Prod to align with broader enterprise patterns and dependencies. Stage is treated as pre-production and is part of the production pipeline promotion model.

### 7) Enterprise Governance Requirement
Production releases require traceability, approvals, coordination, and rollback/backout procedures. Traceability must capture the exact set of tenant revisions included in a production release, plus CCID references.

---

## Decision

Adopt the **Shell Aggregator Pattern** as the long-term build, versioning, and runtime identity strategy.

A single Cloud Manager-connected repository (the Aggregator per ADR-001) is the deployment unit; tenants remain independent in source control, and a controller process assembles tenant source revisions into the Aggregator for each release candidate.

Because Cloud Manager rewrites Maven module versions and OSGi bundle versions in Stage/Prod, **TRP will not rely on Maven version or OSGi `Bundle-Version` as the per-tenant semantic identity in runtime**.

Instead, implement explicit tenant identity through:

1) **Tenant semantic versioning independent of Maven/OSGi**
2) **Runtime discoverability via custom OSGi bundle metadata (headers)**
3) **A release manifest (BOM) that records composition and traceability anchors**

---

## Decision Details

### 1) Source of Truth
* The Aggregator repository is the source of truth for what is deployed.
* The Aggregator commit SHA plus release manifest define the complete release composition.
* Cloud Manager build version identifies the immutable release artifact built and promoted.

### 2) Tenant Semantic Versioning (Independent of Maven/OSGi)
Each tenant maintains a semantic identifier that is not derived from Maven module version or OSGi `Bundle-Version`.

Examples of where this tenant semantic version may live:
* A file such as `TENANT_VERSION`
* A Maven property such as `trp.tenant.semanticVersion`
* A build-time injected value set by the tenant pipeline/controller

This semantic version is intended for:
* cross-team compatibility communication (e.g., “Compose 3.8.0”)
* operational visibility (“what Compose semantic build is running?”)
* audit and change management context

### 3) Runtime Discoverability (Custom OSGi Bundle Headers)
Each tenant bundle embeds explicit identity metadata in bundle headers that are visible in runtime, independent of Cloud Manager rewriting of normal version fields.

Recommended headers:
* `TRP-Tenant-Name`
* `TRP-Tenant-Version` (tenant semantic version)
* `TRP-Git-SHA` (tenant commit SHA)
* Optional: `TRP-Release-Train` or `TRP-Release-ID` (Cloud Manager build/version)

This enables incident response and debugging to determine:
* which tenant semantic versions are present,
* which commits are running,
* without relying on Maven or OSGi version fields that are rewritten.

### 4) Release Manifest (BOM)
A manifest is produced for each release candidate and production release containing:

* Cloud Manager build version (release identifier)
* Aggregator groupId/artifactId and parent version (if needed)
* Aggregator commit SHA
* For each tenant:
  * repo identifier
  * commit SHA
  * tenant semantic version
  * whether it changed in this release
  * CCID references for changed tenants

Storage and retention:
* Manifest is stored in the Aggregator repo (version-controlled)
* Manifest is retained alongside release artifacts for auditability

### 5) Promotion Integrity
Promotion and governance operate on:

* Aggregator commit SHA and tenant SHAs as primary traceability anchors
* Cloud Manager build version as the immutable release identifier
* Tenant semantic versions as functional component identifiers for operational visibility

---

## Rationale

This decision satisfies the constraints simultaneously:

* Single repo input: the Aggregator is the repository connected to Cloud Manager
* Artifactory restriction: code is built from source within the Aggregator; no private binary fetch is required
* Immutable runtime: the Aggregator produces the complete deployable set each time
* Governance: aggregator commit, build id, and manifest provide audit-grade traceability
* Auto-versioning rewrite: tenant identity is preserved via explicit metadata instead of Maven/OSGi version fields

This acknowledges the reality that cross-tenant runtime dependencies become cross-tenant release dependencies in AEMaaCS and optimizes for deterministic composition, repeatability, and defensible governance.

## Options Considered and Why They Were Not Chosen

### Option A: Consume Shared Tenants as Binary Dependencies
Publish shared tenants (e.g., Compose/ECL) as Maven artifacts and have consumers depend on them via Maven dependencyManagement.

Rejected because:
* TRP Artifactory cannot be accessed by Cloud Manager
* Public distribution is typically unacceptable for internal proprietary artifacts
* Increases supply chain and dependency governance complexity
* Does not restore independent runtime deployability; it only shifts coupling earlier in the pipeline

Tradeoff:
May reduce rebuild time, but conflicts with security constraints and does not solve runtime coupling.

---

### Option B: Runtime Sideloading / Plugin Mechanism for Tenant Bundles
Attempt to recreate AMS runtime install behavior via custom mechanisms.

Rejected because:
* Unsupported/high-risk in AEMaaCS
* Violates cloud operational model and may be blocked
* Increases supportability, incident, and security risks

---

### Option C: Use Cloud Manager Build Version as the Only Version
Treat Cloud Manager build identifier as the only version and do not maintain tenant semantic versions.

Rejected because:
* Build version identifies the whole release, not individual tenant components
* Operational debugging and cross-team regression planning requires meaningful tenant-level identifiers
* Incident triage becomes slower and more error-prone

## Risks Introduced by AEMaaCS Constraints (and Why They Are Unavoidable)

### Loss of Independent Tenant Deployability
Risk:
* A change in a shared tenant affects dependent tenants and requires coordinated testing.

Why unavoidable:
* Immutable runtime + Cloud Manager pipeline model prohibit partial runtime installs.

### Increased Blast Radius
Risk:
* A pipeline failure or regression can block all tenants in the deployment unit.

Why unavoidable:
* Cloud Manager gates the whole build unit.

### Slower Rollback
Risk:
* Rollback requires redeploying a previous build (slower than AMS uninstall patterns).

Why unavoidable:
* AEMaaCS does not support uninstall rollback for immutable artifacts.

### Loss of Maven/OSGi Version as Tenant Identity in Stage/Prod
Risk:
* Tenant versions cannot be inferred from Maven coordinates or normal OSGi version fields in runtime.

Why unavoidable:
* Cloud Manager rewrite behavior is enforced and has no module opt-out.

### Potential Version Signal Distortion for Exported Packages
Risk:
* If exported package versions derive from rewritten versions, semantic signaling may be distorted.

Why unavoidable:
* Rewrite is enforced; mitigation requires decoupling semantic identity from rewritten fields.

### Validation Gate Failures Block Promotion
Risk:
* A failure triggered by one tenant can block promotion for all tenants.

Why unavoidable:
* Cloud Manager validation is mandatory and applies to the deployment unit.

---

## Mitigations

* Treat shared tenants as platform providers with explicit compatibility and deprecation policies
* Add integration test coverage in SIT focused on shared dependency consumers
* Enforce release manifest checks as part of promotion and approvals
* Embed tenant semantic version and git SHA into custom bundle headers for runtime validation
* Keep tenant semantic identity independent of Maven/OSGi rewritten fields
* Use staged rollout and progressive validation patterns where possible within AEMaaCS constraints

(Note: operational cadence/governance for release trains and hotfixes is defined in ADR-002.)

---

## Consequences

### Positive
* Complies with Cloud Manager single-repo and immutable runtime model
* Eliminates reliance on inaccessible TRP Artifactory for Cloud Manager builds
* Provides deterministic, auditable release composition via manifest and commit SHAs
* Enables runtime discovery of tenant semantic versions despite version rewriting
* Provides defensible governance aligned with enterprise change control

### Negative
* Loses AMS capability for independent tenant deployments and rapid rollback
* Increases cross-team coordination burden, especially for shared tenant updates
* Increases blast radius of pipeline failures and release regressions
* Requires disciplined metadata and manifest management to preserve tenant identity

---

## Open Questions
* Authoritative process for assigning/approving tenant semantic version increments
* Minimum integration test suite required before promoting shared tenant changes
* Emergency hotfix operational process and rollback time objectives (see ADR-002 for model; define SLOs)
* Whether exported package semantic versions require centralized governance for shared tenants

---

## Notes
This ADR documents constraints introduced by the AEMaaCS platform model that are not fully correctable by process alone. The mitigation is to make release composition explicit, observable, and governed, replacing AMS runtime mutability with deterministic build-and-deploy discipline.
